# Tool5: 获取条目的评论
## 1. 核心意图
[大模型调用此工具的意图]
获取条目的用户评分与短评。用于回答"大家怎么评价这部番""口碑如何"等涉及社区主观意见的问题。

⚠️ **调用前置判断**：条目的客观信息（如剧情简介、声优阵容、Staff）属于 LLM 常识，**无需调用此工具**。此工具仅用于获取 LLM 不可能知道的**实时社区舆论**。

## 2. 聚合的 GET API 路由
1. `[GET] /p1/subjects/{subjectID}/comments` （获取条目吐槽）

## 3. Pydantic 入参建议 (Input)
[大模型调用此工具时需要传入的参数。]
* `subject_id` (int): 条目纯数字 ID。
* `limit` (int, 可选，默认 50，最大值 100): 返回评论条数。通过 `Field(le=100)` 硬性限流。

## 4. 核心 JSON 字段提取 (Output 契约)
**API 1: `/p1/subjects/{subjectID}/comments`**
*(极致瘦身与降噪方案：完全抛弃 JSON 对象，过滤无价值短评，格式化为纯文本列表)*

* `comments` (list[str]): 纯文本吐槽列表。
  * **格式定义**：`"[x星] {comment_text}"` （如果 rate 为 0，则显示为 `[未评分]`）。
  * **示例**：
    * `"[8星] 虽然都是现实中绝对不会遇到的神人，但故事却意外的有真实感..."`
    * `"[10星] 能够家里蹲，本身就是一件奢侈的事..."`

## 5. 业务防御与特殊说明
* **噪音过滤 (Noise Filtering)**：
  必须在 Client 层清洗掉对大模型毫无分析价值的废话，节省 Token。
  * 规则 1：剔除 `comment` 长度小于 4 个字符的评论（过滤掉"补标"、"神作"、"标记"等）。
  * 规则 2：剔除完全由日期、纯数字或标点符号组成的评论（如 "2025-01-28"）。
* **格式极简压缩**：
  直接丢弃 `user`, `id`, `updatedAt`, `type`, `reactions`。不要返回包含 dict 的 list，而是直接拼装成包含 N 个格式化 string 的 list。
* **条目名提示**：在返回 JSON 顶部增加 `"subject": "{条目名}"` 字段，帮助 LLM 快速确认上下文。
* **评分离散分布聚合**：如果 LLM 询问"口碑如何"，可在返回顶部增加 `"rating_distribution": {"1星": N, "2星": N, ...}` 统计摘要供快速判断，避免 LLM 逐条统计。
* **404 异常熔断**：捕获 404，转译为 `"未找到 ID 为 {subjectID} 的条目，可能已被删除或属于隐藏内容"`。
* **空评论处理**：返回 `{"subject": "...", "comments": [], "hint": "该条目暂无社区评论"}`。
* **通用异常熔断**：捕获超时、502、503 等异常，转译为自然语言，绝不允许抛出 Exception 导致状态机崩溃。
