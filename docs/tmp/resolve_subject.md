# Tool4: 条目名称解析（轻量路由）
## 1. 核心意图
[大模型调用此工具的意图]
**仅用于 ID ↔ 名称 的双向解析**。当上游工具（如日历、热门）返回了条目 ID 但缺少中文名，或用户用模糊名称提问需要定位具体条目时调用。

⚠️ **重要前置判断**：如果用户问的是常识性问题（如"奈须蘑菇是谁""Fate/stay night 讲述了什么"），LLM 应直接用自身知识回答，**无需调用此工具**。此工具仅用于需要精确 ID 映射以串联后续 API 调用（如获取评论/章节）的场景。

## 2. 聚合的 GET API 路由
1. `[POST] /p1/search/subjects` （按关键词搜索条目）

## 3. Pydantic 入参建议 (Input)
[大模型调用此工具时需要传入的参数。]
* `keyword` (str): 搜索关键词，通常为条目名称（如 `"进击的巨人"`）或部分名称。
* `limit` (int, 可选，默认 5，最大值 10): 返回结果数。通过 `Field(le=10)` 设置硬性上限，避免搜索返回海量结果。

## 4. 核心 JSON 字段提取 (Output 契约)
**API 1: `/p1/search/subjects`**
*(极致瘦身：仅保留路由所需的最小字段，抛弃 images、info、locked、nsfw、rating 等全部)*

* `results` (list[dict]): 搜索结果列表，每个元素仅包含：
  * `id` (int): 条目纯数字 ID（用于后续 API 调用）。
  * `name` (str): 优先取 `nameCN`，为空则取 `name`。
  * `type` (str): 条目类型。Client 层转为自然语言：`1` → `"书籍"`, `2` → `"动画"`, `3` → `"音乐"`, `4` → `"游戏"`, `6` → `"三次元"`。
* `total` (int): 搜索结果总数，用于判断是否需要缩小搜索范围。

## 5. 业务防御与特殊说明
* **结果为空**：搜索无结果时返回 `{"results": [], "hint": "未找到匹配的条目，请尝试更换关键词或使用日文原名搜索"}`。
* **结果过多**：`total > 50` 时在返回中追加 `"hint": "搜索结果较多，请提供更精确的名称以缩小范围"`，引导 LLM 向用户追问。
* **模糊匹配策略**：搜索接口基于 meilisearch，支持部分匹配。如果 LLM 传入的 `keyword` 过于宽泛（如仅 `"巨人"`），结果可能包含大量无关条目。建议 LLM 调用时尽量使用完整名称。
* **异常熔断**：捕获超时、502、503 等异常，转译为 `"Bangumi 搜索接口响应异常，请稍后重试或尝试使用其他名称"`。
